# 系统监控（System Watch）

## 概述

在每次会话启动时静默执行启动类检测，定时巡检类检测由 cron 自动触发。不替代 CLAUDE.md 中已有的专项检查（如三端路径对齐、依赖文件可达性），而是补充感知盲区。

**七项检测分为两档**（v4.5 起）：

| 档位 | 项目 | 执行时机 | 原因 |
|------|------|---------|------|
| **A 类（启动时）** | #3 断链、#4 记忆冗余、#5 记忆过时、#7 冲突检测 | 每次会话启动 | 只需执行一次即可覆盖 |
| **B 类（定时巡检）** | #1 日志沉寂、#2 草稿积压、#6 回顾提醒 | 每 2 天 cron 自动触发 | >7 天阈值，需定期主动提醒 |

## 监控项

### 1. 日志沉寂

**检测逻辑**：扫描 `2日志与笔记/VibeCoding开发日志/` 下各项目文件夹，读取各项目最新日志的 `updated` frontmatter 字段。

**阈值**：>7 天未更新且项目状态非"完结"或"暂停"。

**提醒**：`📝 项目 <项目名> 已 N 天未写日志，要更新吗？`

**实现**：
```bash
# 扫描项目文件夹，提取 frontmatter 中的 updated 字段
ls -d 2日志与笔记/VibeCoding开发日志/*/ | grep -v _全局规范
# 对每个项目，读取最新 .md 文件的 updated 字段
```

### 2. 草稿积压

**检测逻辑**：扫描 `～草稿箱/` 下所有 .md 文件（含 `手机暂存/` 子文件夹）的修改时间。

**阈值**：>7 天未处理。

**提醒**：`🗂️ 草稿箱有 N 个文件超 7 天未处理，要清理吗？说「整理草稿箱」开始。`

### 3. 断链扫描（精确版）

**检测逻辑**：以 `_系统管理/系统健康/Wiki链接健康报告.md` 为基线，只扫描自上次全量扫描后新增或修改的 .md 文件中的 Wiki 链接。精确版方法：

1. 提取所有 `[[...]]` 链接
2. **排除**：模板目录（`_系统管理/设计规范/笔记格式/`）、代码块内链接、示例文本（`[[A]]`、`[[笔记名]]`、`[[路径/笔记名]]` 等占位符）
3. **分类判定**：
   - 含 `/` 的绝对路径 → 拼接 vault 根路径查找目标文件
   - 不含 `/` 的相对名（如 `[[C5]]`）→ Obsidian 自动按唯一名解析，标记为「相对链接（可解析）」
   - 找不到目标 → 「真断链」
   - 位于历史档案（`_系统管理/系统历史记录/早期记录/`、`v3系列版本/`）= 「历史引文（保留）」
4. 对比基线报告中的已知断链，只报告新增的

**阈值**：任何新增真断链。

**提醒**：`🔗 发现 N 个新增断裂 Wiki 链接：<列出前 3 个>，说「修复断链」处理。`

**实现**：
```bash
# 第一步：获取上次扫描后修改的文件列表
# 第二步：对修改的文件提取 Wiki 链接
# 第三步：排除模板/示例/历史档案
# 第四步：逐个校验目标文件存在性
# 第五步：对比基线报告，报告新增断链
```

> ⚠️ 全量扫描仅在 review 时执行并更新基线报告。日常启动仅做增量扫描。

### 5. 记忆过时检测

**检测逻辑**：扫描各 memory 目录（`~/.claude/projects/<项目>/memory/`）下 .md 文件的 `last-reviewed` frontmatter 字段，与 `_系统管理/系统健康/VERSION.md` 中的当前版本号比较。

**阈值**：任何版本不一致。

**提醒**：`📝 有 N 个 memory 文件可能过时（last-reviewed ≠ vX.X）：<列出文件名>，说「更新历史记录」检查。`

### 7. 三层记忆冲突检测（v4.5 新增）

**检测逻辑**：跨端同名 memory 文件的规则内容对比。若同一规则在两端定义不一致，视为冲突。

1. 读取 Obsidian 端 MEMORY.md：`C:\Users\yrh\.claude\projects\E--2-Notes-0GithubLibraryForObsidian-John-sDataForObsidian\memory\MEMORY.md`
2. 读取 VSCode 端 MEMORY.md：`C:\Users\yrh\.claude\projects\e--6-CodeData\memory\MEMORY.md`
3. 提取两端 memory 索引中的文件名（不含 `.md`），找出同名文件
4. 对每个同名文件，读取两端版本，对比 frontmatter `scope` 字段和核心规则内容
5. 判定冲突：
   - 两端 `scope` 值不同 → ⚠️ 冲突（同一规则不能被同时标记为 L1/L2/L3 的不同层）

   - 两端规则内容一致但表述不同 → ℹ️ 提示可合并
   - 仅一端存在 → 跳过（非冲突）

**阈值**：任何冲突。

**提醒**：`⚠️ 发现 N 个跨端 memory 冲突：<列出文件名 + 冲突类型>，说「修复记忆冲突」处理。`

**实现**：
```bash

# 2. 找出交集（同名文件）
# 3. 逐对读取并对比 frontmatter 的 scope 字段
# 4. scope 不一致 → 报告冲突
# 5. scope 一致但内容可能矛盾 → 标注"需人工判断"
```

> ⚠️ 只检测跨端**同名**文件的冲突。不同名的文件即使规则相似也不标记（可能是有意拆分）。

## 执行流程

### A 类（启动时执行）

1. CLAUDE.md 启动检查触发 system-watch
2. 按顺序执行 A 类检测项（#3 断链、#4 记忆冗余、#5 记忆过时、#7 冲突检测）
3. 静默执行——仅在发现问题时输出提醒
4. 提醒格式统一为 emoji + 简短说明 + 建议触发词

### B 类（定时巡检）

1. 由 cron（`scheduled_tasks.json`，每 2 天 10:17）自动触发
2. 执行 B 类检测项（#1 日志沉寂、#2 草稿积压、#6 回顾提醒）
3. 跨阈值时输出提醒
4. 提醒格式统一为 emoji + 简短说明 + 建议触发词

> cron 创建方式：用户说「设置 system-watch 巡检」→ AI 执行 `CronCreate`

## 边界情况

- **项目文件夹为空**：跳过日志沉寂检测
- **草稿箱为空**：跳过草稿积压检测
- **断链基线不存在**：降级为全量简单扫描，提醒用户执行一次全量扫描
- **MEMORY 文件不存在**：跳过记忆冗余检测
- **memory 目录无 `last-reviewed` 字段**：仅首次提醒，后续静默
- **last-review.txt 不存在**：视为距上次回顾无限大，触发 >14 天提醒
- **检测超时**（>30 秒）：跳过当前项，记录到哨兵
