# 系统设计原则

> Claude VObS（V 和 S 取自 VSCode，Ob 取自 Obsidian）的全景架构说明。修改系统规则前，先读这里。

---

## 一、系统全景

```
┌──────────────────────────────────────────────────┐
│                 三 端 分 工                        │
│                                                   │
│  VSCode (工程师)    Obsidian (编辑)    手机 (秘书)  │
│  写代码 / 调试      管笔记 / 归档      快速存取     │
│  CLAUDE.md          CLAUDE.md          cc-connect │
│  代码工作区\         vault\             → skill 路由│
│         │                │                │       │
│         └────────────────┼────────────────┘       │
│                          │                        │
│              三 层 记 忆（~/.claude/）              │
│         global ──→ project ──→ user              │
│         (全项目通用)  (单项目专属)  (个人偏好)       │
│                          │                        │
│              八 个 Skill                           │
│  quick-capture │ vault-search │ dev-log-creator   │
│  review-dispatcher → review-creator               │
│    archive │ draft-cleanup │ system-watch         │
└──────────────────────────────────────────────────┘
```

**数据流向**：

- VSCode 写代码 → `dev-log-creator` → Obsidian 开发日志
- Obsidian 改架构 → `review-dispatcher` → `review-creator` → 系统历史记录
- 手机发消息 → `quick-capture` / `vault-search` → Obsidian vault
- 对话存档 → `archive` → Obsidian 笔记
- 草稿箱堆积 → `draft-cleanup` → 清理归类
- 系统健康 → `system-watch` → 启动时 A 类检测 + B 类定时巡检 cron
- 跨系统对齐靠 `project` frontmatter 字段 + 同名文件夹

## 二、Vault 内容架构

系统层之外，vault 内部有一套面向**知识成熟度**的组织逻辑：

| 层 | 文件夹 | 内容状态 | 典型例子 |
|----|--------|---------|---------|
| 入口层 | `～草稿箱/` | 未消化 | 临时想法、外部论文链接、手机暂存 |
| 加工层 | `1主题研究/` · `2日志与笔记/` | 加工中 | 深读判断（想→写）、过程记录（做→记） |
| 交付层 | `3工作与产出/` | 已完成 | 可对外、可提交的成品 |

流向是单向的：入口层 → 加工层 → 交付层。

### Architecture A：草稿箱作为唯一入口

所有外部资料（论文、GitHub 项目、他人文章）统一从 `～草稿箱/` 进入 vault。存入时 AI 自动写入 `source`（原始链接）和 `prerequisites`（前置知识清单）两个字段。`unread → active → archived` 状态机追踪消化进度。

**设计原则**：外部信息和内部知识之间有明确的边界，跨越边界需要人的消化。AI 可以帮忙整理，但不能自动归档外部资料到主题文件夹。

### 自说明体系

四个核心文件夹各含 `README.md`，形成自说明链——打开任意文件夹，第一个文件解释「这是什么、什么该放这里、什么不该」。AI 和人都能在不读 CLAUDE.md 的情况下理解文件夹用途。

设计原则的外化：**结构本身就是文档**。

## 三、核心设计决策

### 3.1 八大决策

| 决策 | 内容 | 设计理由 |
|------|------|---------|
| **角色分化** | VSCode = 工程师 / Obsidian = 编辑 / 手机 = 秘书 | 每端只做自己擅长的事。CLAUDE.md 不相互包含，减少 token 消耗 |
| **内容成熟度三层** | 草稿箱（入口）→ 主题研究/日志（加工）→ 工作产出（交付） | 按知识消化程度分层，流向单向。外部信息进草稿箱，消化后升级 |
| **自说明结构** | 每个文件夹自带 README | 结构本身就是文档。打开文件夹就知道放什么 |
| **三层记忆** | global → project → user | `scope` 字段区分层级。project 级记忆随项目完结可清理，防止无限膨胀 |
| **跨系统对齐** | `project` 字段 + 同名文件夹 | 纯命名约定，不依赖 API 或数据库。机械匹配，不走 AI 猜测 |
| **Skill 管道** | 8 个 Skill，自然语言触发词路由 | CLAUDE.md 是路由表（唯一权威），SKILL.md 是执行手册。触发词只在一处枚举 |
| **版本化演进** | 语义化版本 + 历史档案原文不动 | 小修补合并到当前版本。旧档案不动，结尾加对照表建立新旧概念链接 |
| **单源权威** | 每类信息只有一个权威来源 | 触发词路由 → CLAUDE.md，操作步骤 → SKILL.md，跨端规则 → 跨端共享规范。不重复 |

### 3.2 设计密度

> 设计密度的定义：**不是文件多、代码多。是每做一个设计决策，解决了多少个问题。**

以 `project` 字段为例——一行 YAML frontmatter：

```yaml
project: 01-德州扑克-20260606
```

这一行同时解决了四个问题：

| # | 作用 | 说明 |
|:--:|------|------|
| 1 | **跨端精确关联** | VSCode 和 Obsidian 两端无需 API、无需数据库 |
| 2 | **日志自动归位** | 新项目创建时日志不会写错文件夹 |
| 3 | **记忆可清理** | 项目完结后 project 级记忆可识别、可清理 |
| 4 | **跨会话可追溯** | 任何时候都知道一条日志属于哪个项目 |

## 四、三层记忆架构

三层记忆解决的核心问题：**AI 每次对话从零开始——它不知道你是谁、在做什么、偏好什么风格。**

```
所有项目都适用的规则    → global   (全局)
只跟某个项目有关的规则  → project  (项目)
个人偏好和习惯         → user     (用户)
```

每层通过 `scope` frontmatter 字段标记，加载时按 global → project → user 优先级叠加。MEMORY.md 采用 L1/L2/L3 三层分组索引（等价于目录隔离，零迁移成本）。27 个 memory 文件已完成 `scope` + `last-reviewed` 根级标准化。

### 与工业系统的同构

| 系统 | 分层模型 |
|------|---------|
| Kubernetes | Cluster → Namespace → Pod |
| Unix 配置 | `/etc/` → `~/.config/` → `./.env` |
| Claude VObS | global → project → user |

这不是巧合。所有需要同时处理"全局约束 + 局部特化 + 个人偏好"的复杂系统，分层是唯一收敛的解。

## 五、五层逻辑纵深

### 5.1 框架

```
用户看到的 ────→  交互逻辑（什么时候说什么、触发什么）
                  │
系统做的事 ────→  业务逻辑（触发之后，执行什么规则）
                  │
数据怎么走 ────→  数据逻辑（信息存在哪、怎么流转、谁读谁写）
                  │
出事了怎么办 ──→  容错逻辑（断了怎么接、错了怎么修）
                  │
系统怎么长大 ──→  演进逻辑（怎么记录变化、怎么避免变烂）
```

### 5.2 Claude VObS 逐层评估

| 逻辑层 | 评分 | 说明 |
|--------|:--:|------|
| 交互逻辑 | ⭐⭐⭐⭐ | 14 条意图路由 + 意图匹配 + 三级优先级（v4.4 起 #3-#5 合并为统一入口）。不要求精确匹配，按意图判断 |
| 业务逻辑 | ⭐⭐⭐⭐⭐ | 8 Skill 边界清晰，每条管线含完整的执行步骤、边界情况和错误处理 |
| 数据逻辑 | ⭐⭐⭐⭐⭐ | `project` 字段写入前自动校验；三层记忆按 scope 分层管理；会话接力双写形成冗余；`arch-changes.jsonl` VSCode→Obsidian 架构变更管道 |
| **容错逻辑** | ⭐⭐⭐ | 8 条流水线全覆盖哨兵追踪 + 降级本地缓存 + 启动自动检测 + Git 安全网 + B 类定时巡检。但自动修复回路仍未闭环 |
| 演进逻辑 | ⭐⭐⭐⭐⭐ | 语义化版本 + 历史档案不动 + 📌 对照表 + `last-reviewed` 过时检测 + 定期提醒，完整自我迭代闭环 |

## 六、韧性评估

### 6.1 韧性工程四能力

| 能力 | 含义 | 当前水平 | 说明 |
|------|------|:--:|------|
| 吸收 (Absorb) | 出事了能不崩吗？ | ⭐⭐⭐ | 8 条管线均有降级路径，vault 不可访问时自动切本地缓存 |
| 适应 (Adapt) | 崩了能换条路走吗？ | ⭐⭐ | 降级路径全面，但切换后仍需人工决策下一步 |
| 恢复 (Recover) | 崩完能自己修好吗？ | ⭐⭐ | 哨兵文件完整追踪断裂点 + 启动自动检测 + Git 安全网回滚，但恢复仍需人说触发词 |
| 学习 (Learn) | 崩过能不再崩吗？ | ⭐⭐⭐⭐ | 版本化历史 + `last-reviewed` 过时检测 + `system-watch` 五项监控，每次断裂后规则可升级 |

### 6.2 反馈闭环开放项

- **主动感知** ✅：`system-watch` A 类启动时 + B 类 cron 定时巡检（v4.5 起每 2 天自动执行），覆盖日志沉寂、草稿积压、断链、记忆冗余、记忆过时、回顾提醒、三层记忆冲突检测共七项
- **自动修复** 🟡：确定的低风险问题可自动修复 + 告知，无需等待用户触发词（v4.5 已讨论，待确认执行边界）
- **冗余冲突检测** ✅：三层记忆冲突检测已纳入 `system-watch` 第 7 项（v4.5 起）
- **记忆过时检测** ✅：`last-reviewed` 版本标记机制已全面覆盖 27 个 memory 文件，`system-watch` + `review-creator` 双轨检测
- **Token 优化** ✅：估算方法升级为输入轮数×2000 + 输出字节×1.5，模型路由三档分级（LIGHT/STANDARD/HEAVY，v4.5 起）

## 七、项目生命周期

完整闭环：新建 → 日常开发 → 完结 → 完结后维护。

| 阶段 | VSCode 端动作 | Obsidian 端动作 |
|------|-------------|----------------|
| 新建 | 建项目文件夹 + 源码 | 建日志文件夹 + 首个 .md |
| 日常开发 | 写代码，说「提交」一次触发 commit+push+日志 | 接收日志写入 |
| 完结 | 最终验证 + 创建回顾总结 | 归档项目记忆 |
| 完结后维护 | Bug 修复，自动追加维护记录 | 维护记录追加到项目总结 |

跨端通过 `project` 字段 + `arch-changes.jsonl` 管道同步状态。

## 八、评估模式

评审报告可在 frontmatter 中声明 `eval_mode` 字段：

| 模式 | 含义 | 行为 |
|------|------|------|
| `framework-first` | 系统在框架搭建阶段，内容填充不是当前重点 | 内容产出维度跟踪但不评分 |
| （不声明） | 默认全维度评估 | 所有维度参与评分 |

## 九、设计约束

修改系统规则时的检查清单：

- [ ] **是否破坏了单源权威？** 新内容是否只在一处定义？
- [ ] **两端是否需要同步？** 改了 Obsidian 端规范，VSCode 端的引用路径是否还正确？
- [ ] **历史档案处理原则**：旧版本文件不动，结尾加 📌 对照表建立新旧概念链接
- [ ] **Skill 触发词**：是否只在 CLAUDE.md 中枚举？
- [ ] **project 字段**：两端 project 值是否一致？
- [ ] **跨端触发词无歧义？** "推送" ≠ "发布"，「提交」的端归属是否清晰？

## 十、已知局限

| 局限 | 影响 | 改进方向 |
|------|------|---------|
| 容错回路不闭环 | 流水线断裂后无法自愈，需人工说触发词恢复。v4.5 已实现 system-watch 定时巡检主动发现，但自动修复仍需确认执行边界 | 定期自动重试 + 失败升级通知 |
| 单用户设计 | 无多用户隔离和权限模型 | 短期不需要；多用户时需架构升级 |
| 跨端同步靠命名约定 | `project` 字段已自动修正，但文件夹名不一致时仍需人工介入 | 扩展自动校验到文件夹名层面 |
| 路径硬编码 | 换机器或换用户时需手动替换占位符 | 考虑配置文件统一管理路径 |
