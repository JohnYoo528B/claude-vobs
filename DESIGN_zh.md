# 系统设计原理

> Claude VObS 的架构原理。本文档是系统设计决策的**唯一权威来源**。
> 当前适用版本：**v4.4 起** · 系统当前版本见 [[_系统管理/系统健康/VERSION]] · 旧版快照见 [[原理旧版/系统设计原理-v4.0-v4.4]] · 更早见 [[原理旧版/系统设计原理-v3.7-v4.0]]

## 一、系统本质

Claude VObS 不是笔记工具，不是代码助手，不是 AI 问答前端。它是一套 **以 AI 为内核的个人操作系统**。

这个判断基于结构同构，而非修辞类比：

| OS 组件 | 传统操作系统 | Claude VObS |
|---------|------------|-------------|
| 内核 | Linux Kernel | Claude Code CLI |
| 进程调度 | CPU scheduler | Skill 路由表（触发词 → 执行管道） |
| 文件系统 | ext4 / NTFS | Markdown + YAML frontmatter + Wiki-link |
| 内存管理 | RAM + swap | 三层记忆（global → project → user） |
| 用户空间 | Shell / GUI | VSCode（工程）+ Obsidian（知识）+ 微信（捕捉） |
| 设备驱动 | Kernel modules | cc-connect 手机端适配 |
| 日志系统 | syslog / journald | dev-log-creator + review-creator |
| 包管理 | apt / npm | Skill 体系（可插拔执行模块） |

**核心特征**：AI 在不同端点扮演不同角色、遵循不同规则、操作不同数据。人不写代码——人设计规则。AI 在规则内执行。

### 1.1 判定链：OS 的核心操作

如果 Claude VObS 是一台操作系统，它的指令周期就是**判定**。整个系统的运作可以抽象为一条逐层推进的分类管道——每层是一个分类器，输入被逐层分类，分类结果决定路由目标。

```
用户输入
  │
  ▼
┌─────────────────┐
│ ① 意图判定       │  → 「想干什么？」
│ 分类器：路由表    │     意图类型 → 选择 Skill / 执行管道
└────────┬────────┘
         ▼
┌─────────────────┐
│ ② 来源判定       │  → 「从哪来的？」
│ 分类器：端点标识  │     VSCode / Obsidian / 手机 → 赋予对应角色行为
└────────┬────────┘
         ▼
┌─────────────────┐
│ ③ 类型判定       │  → 「什么性质的操作？」
│ 分类器：Skill 管道│     捕捉 / 搜索 / 日志 / 回顾 / 归档 → 选择执行规则
└────────┬────────┘
         ▼
┌─────────────────┐
│ ④ 成熟度判定     │  → 「内容有多成熟？」
│ 分类器：状态机    │     unread → active → archived → 决定存放位置
└────────┬────────┘
         ▼
┌─────────────────┐
│ ⑤ 归属判定       │  → 「属于哪个项目？」
│ 分类器：project 字段│  跨端一致标识 → 路由到正确的文件夹/日志/记忆层
└────────┬────────┘
         ▼
┌─────────────────┐
│ ⑥ 影响判定       │  → 「改多大？」
│ 分类器：量化标准  │     大改/中改/小修/跳过 → 决定回顾深度和传播范围
└────────┬────────┘
         ▼
      执行
```

六层判定链穿了系统全部五层逻辑纵深（§五）：

| 判定层 | 对应的逻辑纵深 | 判定的东西 | 路由结果 |
|--------|:--:|------|------|
| ① 意图 | 交互逻辑 | 用户想做什么 | 14 条路线之一 → Skill |
| ② 来源 | 交互逻辑 | 从哪个端进入 | 角色行为宪法（工程师/编辑/秘书） |
| ③ 类型 | 业务逻辑 | 操作属于哪类 | 5 个 Skill 之一 → 执行管道 |
| ④ 成熟度 | 数据逻辑 | 内容消化到什么程度 | 草稿箱 / 主题研究 / 工作与产出 |
| ⑤ 归属 | 数据逻辑 | 属于哪个项目域 | project 字段 → 文件夹 + memory 层 |
| ⑥ 影响 | 演进逻辑 | 改动触及什么层面 | 完整更新 / 微调 / 收工 / 跳过 |

**关键洞察**：系统设计原理 §二-§六 描述的是各个组件的**静态结构**（三层架构是什么、Skill 有哪些、容错怎么设计），而判定链描述的是这些组件**如何协同运作**——输入如何从用户的自然语言开始，经过六次逐层判定，最终变成精确的执行动作和正确的存储位置。

这个模式的通用性超出了 Claude VObS 本身：任何复杂 AI 系统，只要需要处理多种类型的输入并路由到多种执行路径，都可以用「判定链」建模。判定链的本质是**把不确定性逐层消解**——用户的一句话在最上层是模糊的（"整理一下"），经过六层判定后在最下层是精确的（创建文件 → 写入 YAML frontmatter → 更新 Wiki 链接 → 标记 consumed）。

**设计含义**：新增功能时，问的不是"这东西放哪个文件夹"，而是"它在判定链的哪一层、作为什么分类器"——这比"看哪个文件夹顺眼就放哪"的直觉式设计多了一层结构保证。

## 二、三层架构

### 2.1 架构来源

三层架构不是理论推导的产物，而是从具体痛点中浮现的：

```
问题：AI 每次对话从零开始 → 写进 CLAUDE.md
发现：不是所有东西都该写进 CLAUDE.md →
      有些只属于某个项目，有些只是个人偏好 →
      本能地分了层
```

这个思维路径与 Unix 的 `/etc/` → `~/.config/` → `./.env`，以及 Kubernetes 的 Cluster → Namespace → Pod 完全同构。三个系统彼此独立，独立发现了同一条规律——**说明分层是复杂系统的收敛解，不是偶然**。

### 2.2 三层定义

| 层级 | 范围 | 存放 | 生命周期 |
|------|------|------|---------|
| **端层**（角色分化） | VSCode / Obsidian / 手机 各自独立的行为宪法 | 各端 CLAUDE.md | 永久 |
| **记忆层**（知识持久化） | global → project → user 三级记忆 | `~/.claude/projects/` | project 级随项目完结可清理 |
| **管道层**（执行编排） | 4 个 Skill 按触发词路由 | `~/.claude/skills/` | 可插拔 |

### 2.3 设计密度

高设计密度意味着单个决策解决多个问题。范例——`project` 字段：

```yaml
project: 01-德州扑克-20260606
```

一行 YAML，同时解决：

| # | 作用 |
|:--:|------|
| 1 | 跨端精确关联（VSCode memory ↔ Obsidian 日志，无需 API） |
| 2 | 日志自动归位（dev-log-creator 不会写错文件夹） |
| 3 | 记忆可清理（项目完结后 project 级 memory 可 GC） |
| 4 | 跨会话可追溯（任何时候知道某日志属于哪个项目） |

**评价标准**：不是文件多、代码多。是每做一个决策解决了多少问题。

## 三、Vault 内容架构

系统层之外，vault 内部还有一套内容组织逻辑——同样是三层，但面向的是**知识的成熟度**而非系统功能。

### 3.1 内容成熟度三层

| 层 | 文件夹 | 内容状态 | 典型例子 |
|----|--------|---------|---------|
| 入口层 | `～草稿箱/` | 未消化 | 临时想法、外部论文链接、手机暂存 |
| 加工层 | `1主题研究/` · `2日志与笔记/` | 加工中 | 深读判断（想→写）、过程记录（做→记） |
| 交付层 | `3工作与产出/` | 已完成 | 可对外、可提交的成品 |

流向是单向的：入口层 → 加工层 → 交付层。一条信息从草稿箱出发，经过消化加工，最终要么变成自己的判断（主题研究），要么变成成品（工作与产出）。

### 3.2 Architecture A：草稿箱作为唯一入口

v4.0 确立：所有外部资料（论文、GitHub 项目、他人文章）统一从 `～草稿箱/` 进入 vault。不再有 `_外部参考/` 之类分散的子目录。

这是一个**系统性约束变更**——它不是在管理文件夹，而是在确立一条设计原则：**外部信息和内部知识之间有明确的边界，跨越边界需要人的消化**。AI 可以帮忙整理，但不能自动归档外部资料到主题文件夹。

### 3.3 外部资源状态机

```
unread → active → archived
```

| 状态 | 含义 | 触发 |
|------|------|------|
| `unread` | 还没读 | 外部资料首次存入，AI 写入 frontmatter |
| `active` | 正在学 | 用户说「学了 xxx」后自动切换 |
| `archived` | 已消化 | 消化完成，原始文件删除，内容升级为主题笔记 |

存入时 AI 自动写入 `source`（原始链接）和 `prerequisites`（前置知识清单）两个字段。启动时按前置知识难度分级提醒未读资源。

### 3.4 README 自说明体系

四个核心文件夹（`～草稿箱/`、`1主题研究/`、`2日志与笔记/`、`3工作与产出/`）各含 `README.md`，形成自说明链：

- 打开任意文件夹，第一个文件解释「这是什么、什么该放这里、什么不该」
- AI 和人都能在不读 CLAUDE.md 的情况下理解文件夹用途
- 每个 README 的核心问题是「判断标准」——读者能据此自己决定内容归属

这是一个设计原则的外化：**结构本身就是文档**。文件夹的组织逻辑不应该只在 CLAUDE.md 里描述，而应该嵌入结构本身。

## 四、核心设计原则

### 4.1 角色分化（Role-Based Access Control）

同一 AI 在三端赋予三种行为人格。每端的 CLAUDE.md 是独立的 behavioral policy file。这在分布式系统领域对应 **RBAC** 和 **Context-Aware Behavior**。

| 端点 | 角色 | 核心约束 | 领域映射 |
|------|------|---------|---------|
| VSCode | 工程师 | Plan → Code → Explain | 开发环境 |
| Obsidian | 编辑 | 读 → 搜 → 关联，禁止主动写代码 | 知识管理 |
| 手机 | 秘书 | 捕捉 → 确认，一字别多 | 低摩擦入口 |

### 4.2 跨系统对齐（Shared Identifier Strategy）

VSCode 项目文件夹名 = Obsidian 日志文件夹名。`project` frontmatter 字段值两端一致。

这在分布式系统里对应 **Correlation ID** 和 **Event Sourcing**——在没有 API、消息队列、数据库的情况下，用命名约定 + frontmatter 字段手工实现了跨系统事件溯源。

### 4.3 单源权威（Single Source of Truth）

每类信息只有一个权威来源：

| 信息类型 | 权威来源 |
|---------|---------|
| 触发词路由 | CLAUDE.md |
| 执行步骤 | SKILL.md |
| 设计原理 | 本文档 |
| 跨端规范 | `跨端共享规范.md` |
| 项目规范 | `_全局规范/` |

规则不在两处同时定义。引用对端信息时通过 CLAUDE.md 的关联系统节链过去，不复制粘贴。

### 4.4 版本化演进（Semantic Versioning + Database Migration）

- 大版本（架构重组 / 格局变化）→ 主版本号 +1
- 功能新增 → 次版本号 +0.1
- 小修小补 → 合并到当前版本，不开新号

历史档案处理原则——**与数据库 migration 哲学一致**：旧版本文件原文不动，结尾加 📌 对照表建立新旧概念的桥接。旧版使用指南归档到历史记录末尾后删除。

## 五、五层逻辑纵深

系统完整逻辑从用户可见的表层到自我演进的基础层，共五层：

```
用户看到的 ────→  交互逻辑（什么时候说什么、触发什么）
                  │
系统做的事 ────→  业务逻辑（触发之后，执行什么规则）
                  │
数据怎么走 ────→  数据逻辑（信息存在哪、怎么流转、谁读谁写）
                  │
出事了怎么办 ──→  容错逻辑（断了怎么接、错了怎么修）
                  │
系统怎么长大 ──→  演进逻辑（怎么记录变化、怎么避免变烂）
```

### 5.1 交互逻辑

触发词体系 + 意图匹配 + 三级优先级。不要求精确匹配——用户说自然语言，AI 按意图判断。

14 条意图路线覆盖全部操作。v4.4 将「历史记录/微调/收工」三条合并为「系统回顾（统一入口）」→ review-dispatcher，路由表从 16 行缩减为 14 行。

### 5.2 业务逻辑

5 个 Skill 构成管道层，遵循单一职责原则。v4.4 起 review-dispatcher 位于 review-creator 之前，实现四个回顾入口的统一分流：

| Skill | 触发场景 | 职责边界 |
|-------|---------|---------|
| quick-capture | 手机快速捕捉 | 原文照存，只回 ✅ |
| vault-search | vault + 对话记忆并发搜索 | 分级回复（1-5 逐条 / 6-15 列名 / >15 给前 5） |
| dev-log-creator | 代码变更 → Obsidian 日志 | 先问用户做了什么，不盲目扫代码 |
| review-dispatcher | 系统回顾统一入口（v4.4 新增） | 四路径判定（完整更新/微调/收工/自动提醒）→ 参数打包 → 调用 review-creator |
| review-creator | 架构变更 → 历史记录 + 指南更新 | 8 Phase 参数化引擎（v4.3 为 15 步固定流程，v4.4 加参数接口+追加模式，v4.5 加 max_tokens 预估+memory last-reviewed 自动更新） |

> v4.4 之前（v4.0-v4.3）为 4 个 Skill——review-dispatcher 不存在，四个回顾入口各自维护不完整逻辑。v4.4 起统一为 dispatcher → creator 的二级管道架构。
> v4.5 新增 system-watch B 类定时巡检 cron + 第 7 项三层记忆冲突检测 + memory frontmatter 标准化（scope/last-reviewed 根级约束），见 [[memory-frontmatter-标准]]。

### 5.3 数据逻辑

三层记忆 + `project` 字段 + scope frontmatter 自动路由。

- L1 global：所有项目通用，常驻上下文
- L2 project：单项目专属，按需加载，项目完结可 GC
- L3 user：个人偏好，影响回答风格

数据流方向严格单向：手机 → 草稿箱 → 归档；VSCode → dev-log → Obsidian；Obsidian → review → 历史记录。

> 完整项目生命周期（新建→日常→完结→完结后维护）见 [[跨端共享规范]] §七。

### 5.4 容错逻辑

四层韧性机制：

| 机制 | 对应韧性能力 | 实现 |
|------|:--:|------|
| 全局重试 + 超时 | 吸收（Absorb） | 失败 → 等 2s → 重试 1 次 |
| 静默降级（fallback 目录） | 适应（Adapt） | vault 不可访问 → 本地缓存 |
| 哨兵恢复 + Git 安全网 | 恢复（Recover） | 哨兵记录 → 恢复指令 → 自动重试 |
| 定期审计 + 自检 | 学习（Learn） | 启动健康检查 + 韧性自检清单 |

统一哨兵表覆盖 7 条流水线。Git 安全网在每次写操作前自动快照，失败可 `git reset --hard` 回滚。

### 5.5 演进逻辑

系统本身有自我迭代机制：版本化历史记录 + 架构变更追踪 + 定期回顾提醒 + 使用指南同步更新。每个版本有明确的"变更了什么、卡在哪里、学到了什么"。

## 六、反馈与控制

### 6.1 控制论视角

系统当前具备**传感器**（review-creator 的磁盘/Token/项目扫描），缺少**自动反馈回路**。检测到问题后仍需要人说触发词才执行修复。从控制论角度看，这是一个**人在回路中的半闭环系统**。

完整闭环应是：传感器检测 → 判断严重程度 → 低风险自动修复 + 告知 / 高风险提醒用户确认 → 执行 → 记录结果。当前停留在第一步。

### 6.2 韧性工程

| 能力 | 含义 | 状态 |
|------|------|:--:|
| 吸收（Absorb） | 出事了能不崩 | ✅ |
| 适应（Adapt） | 崩了能换条路 | ✅ fallback 目录 |
| 恢复（Recover） | 崩完能自己修 | ✅ 哨兵 + Git 回滚 |
| 学习（Learn） | 崩过能不再崩 | ✅ 启动自检 + 审计 |

### 6.3 反馈闭环开放项

| 项目 | v4.4 状态 | v4.5 进展 |
|------|:--:|------|
| 主动感知 | ❌ 5 个 Skill 全部被动响应 | 🟡 system-watch B 类定时巡检（cron 每 2 天），覆盖日志沉寂/草稿积压/回顾提醒 |
| 自动修复 | ❌ 需等用户触发词 | ❌ 仍为开放项 |
| 冗余冲突检测 | ❌ 三层记忆之间无冲突检测 | ✅ system-watch #7 跨端同名 memory 冲突检测（v4.5） |
| 记忆过时检测 | 🟡 v4.1 启动双轨检测，但 memory 字段不统一 | ✅ scope + last-reviewed 全覆盖 27 个 memory 文件，review-creator 14d 自动批量更新 |
| Token 估算精度 | ❌ 轮数×经验值粗估 | 🟡 升级为「输入轮数×2000 + 输出字节×1.5」+ method 字段标注 |
| 模型选择 | ❌ 无规则，默认用同一模型 | 🟡 三档任务分级 + max_tokens 按需设定（模型路由规则） |

**仍为开放项**：自动修复（需确认执行边界后才能推进）、智能模型自动切换（受限于 Claudian 会话级模型选择）

## 七、不可删除的核心

如果删到只剩骨架，系统仍成立的最小集合：

```
CLAUDE.md（行为宪法）
    +
project 字段（跨系统识别码）
    +
三层记忆（global → project → user）
```

这三者构成了系统的最小完整定义。其余一切（Skill、触发词表、启动检查、哨兵降级、历史记录）都是这三者之上的增量。

## 八、系统边界与约束

### 单点依赖

- 两份 CLAUDE.md 定义了全部 AI 行为。损坏一份 → 对应端点 AI 退化到无上下文状态
- 手机端依赖 cc-connect 外部组件

### 外部依赖

- AI 模型（DeepSeek V4 Pro，通过 Anthropic 兼容 API）
- Python 环境（conv-search.py 对话记忆搜索）
- GitHub（公开版发布）

### 设计检查清单

修改系统规则前，确认以下各项：

## 九、系统定位：四层架构

> v4.9 收工讨论中提出。本节从"VObS 到底是什么"出发，解剖系统的四层结构，并附 Hermes Agent 外部对标验证。
> 本节是 §一「系统本质」的深化——§一 用 OS 隐喻描述结构，本节用四层解剖描述品类。

### 9.1 为什么需要定位

"Agent""Cowork""操作系统""第二大脑"——每个词都只能覆盖 VObS 的一部分。Agent 语境已坍缩为"全自动"（VObS 核心是人在回路）；Cowork 太泛（Kimi/ChatGPT 都叫 cowork）；操作系统是隐喻不是定义。

需要一个**精确的、可拆解的定位**——不是一句话 slogan，而是一个能区分"这属于哪一层"的分析框架。

### 9.2 四层解剖

| 层 | 核心问题 | VObS 的实现 | 类比 |
|------|------|------|------|
| **判定引擎** | "怎么判断该做什么" | Markdown 规则驱动——AI 自己读规范文件、自己判定、自己走流程（不是代码 if-else） | 员工手册——不是硬编码的自动化脚本，是人能看懂、AI 能执行的规则文本 |
| **自我组织** | "系统怎么管理自己的演进" | PENDING-CHANGES + review-dispatcher/creator + VERSION + 下游传播——系统追踪自己的变化、生成自己的历史 | 公司内控——不是外部审计师，是系统内部自带的记录、回顾、改进机制 |
| **外部连接** | "怎么触达外面的世界" | Playwright MCP + WebSearch + cc-connect + ntfy.sh + GitHub API——可插拔的能力中枢，需要什么接什么 | 工具箱——不是内置全部能力，是按需接入的标准接口 |
| **实操侧重** | "人怎么和系统交互" | 半自动协同：人说→AI 提案→人确认→AI 执行→自动收尾——人不让出控制权 | 副驾驶——方向盘在人手里，AI 负责油门刹车导航 |

### 9.3 四层之间的关系

```
实操侧重（人怎么交互）
    ↕ 驱动
判定引擎（怎么判断该做什么）
    ↕ 产出：路由结果 + 影响判定
自我组织（怎么管理演进）
    ↕ 调用
外部连接（怎么触达世界）
```

四层不是平级的——判定引擎是核心，自我组织是它驱动的"内循环"，外部连接是它调用的"外循环"，实操侧重是它的人机界面。

### 9.4 综合定位

> **Claude VObS 是一个「Markdown 规则驱动的 AI 协同系统」**

拆解这个定义：

| 词 | 含义 | 对应层 |
|------|------|:--:|
| Markdown 规则 | 系统行为由人可读、人可改的 Markdown 文件定义（不是代码、不是向量、不是 GUI 配置） | 判定引擎 |
| 驱动 | 规则不是参考——是 AI 实际执行的指令。AI 自己读、自己判定、自己走流程 | 判定引擎 |
| AI | Claude 是执行内核，但系统本身不绑死单一模型 | 外部连接 |
| 协同 | 不是全自动 agent，是半自动——人提案→AI 执行→人确认→自动收尾 | 实操侧重 |
| 系统 | 不只是工具集合——有自我组织能力（记录变化、生成历史、管理版本） | 自我组织 |

### 9.5 四层定位的区分力

拿这个框架去看市面上的产品，能精确区分"它有什么、它缺什么"：

| 产品 | 判定引擎 | 自我组织 | 外部连接 | 实操侧重 |
|------|:--:|:--:|:--:|:--:|
| **ChatGPT** | ❌ AI 隐式判断 | ❌ 无 | 🟡 插件/Web | 🟡 对话 |
| **Kimi** | ❌ AI 隐式判断 | ❌ 无 | ✅ 浏览器/Web | 🟡 对话 |
| **Cursor** | 🟡 .cursorrules | ❌ 无 | ✅ LSP/终端 | ✅ 半自动编辑 |
| **Hermes Agent** | 🟡 Python + SKILL.md | ✅ Honcho + 自动 Skill | ✅ MCP + 消息平台 | ✅ 全自动 agent |
| **VObS** | ✅ Markdown 规则驱动 | ✅ review + VERSION | ✅ MCP + cc-connect + ntfy | ✅ 半自动协同 |

> VObS 是唯一一个四层全绿的系统。Hermes 缺在判定引擎（规则主体是代码），Cursor 缺在自我组织，ChatGPT/Kimi 缺在前两层。

### 9.6 Hermes 对标：殊途同归

Hermes Agent（Nous Research 开源，GitHub 17 万星）是市面上与 VObS 最接近的系统——Skill 体系（`~/.hermes/skills/` SKILL.md）、多层记忆（四层向量 DB）、自我复盘（成功→自动封装 Skill）、跨端连续（Telegram/终端/微信）、定时任务（cron）——五个子系统与 VObS 高度同构。

但搭建方式完全相反：**Hermes 是用 Python 代码搭的→后来补了 Skill Markdown 能力；VObS 是用自然语言 + Markdown 搭的→后来接了 MCP、加了 cron。** 同一个问题的同一个解——方向相反，结论一样。

详见 [[v4.9评审报告-202606130320]] §二。

### 9.7 不可替代性

市面上的 AI 工具各自覆盖四层中的 1-2 层。VObS 的不可替代性不在于"哪一层做得最好"，而在于**四层同时存在、且判定引擎是用人能看懂的 Markdown 实现的**——这意味着一个非程序员可以自己搭建、自己修改、自己演进。

这不是"比 Hermes 更好"——这是"给不会写代码的人一个 Hermes 级别的系统"。
