# 系统设计原则

> Claude VObS（Claude + VSCode + Obsidian + System）的全景架构说明。修改系统规则前，先读这里。

---

## 一、系统全景

```
┌──────────────────────────────────────────────────┐
│                 三 端 分 工                        │
│                                                   │
│  VSCode (工程师)    Obsidian (编辑)    手机 (秘书)  │
│  写代码 / 调试      管笔记 / 归档      快速存取     │
│  CLAUDE.md          CLAUDE.md          cc-connect │
│  代码工作区\         vault\             → skill 路由│
│         │                │                │       │
│         └────────────────┼────────────────┘       │
│                          │                        │
│              三 层 记 忆（~/.claude/）              │
│         global ──→ project ──→ user              │
│         (全项目通用)  (单项目专属)  (个人偏好)       │
│                          │                        │
│              四 个 Skill                           │
│  quick-capture │ vault-search │ dev-log-creator   │
│                         │ review-creator          │
└──────────────────────────────────────────────────┘
```

**数据流向**：

- VSCode 写代码 → `dev-log-creator` → Obsidian 开发日志
- Obsidian 改架构 → `review-creator` → 系统历史记录
- 手机发消息 → `quick-capture` / `vault-search` → Obsidian vault
- 跨系统对齐靠 `project` frontmatter 字段 + 同名文件夹

## 二、核心设计决策

### 2.1 六大决策

| 决策 | 内容 | 设计理由 |
|------|------|---------|
| **角色分化** | VSCode = 工程师 / Obsidian = 编辑 / 手机 = 秘书 | 每端只做自己擅长的事。CLAUDE.md 不相互包含，减少 token 消耗 |
| **三层记忆** | global → project → user | `scope` 字段区分层级。project 级记忆随项目完结可清理，防止无限膨胀 |
| **跨系统对齐** | `project` 字段 + 同名文件夹 | 纯命名约定，不依赖 API 或数据库。机械匹配，不走 AI 猜测 |
| **Skill 管道** | 4 个 Skill，自然语言触发词路由 | CLAUDE.md 是路由表（唯一权威），SKILL.md 是执行手册。触发词只在一处枚举 |
| **版本化演进** | 语义化版本 + 历史档案原文不动 | 小修补合并到当前版本。旧档案不动，结尾加对照表建立新旧概念链接 |
| **单源权威** | 每类信息只有一个权威来源 | 触发词路由 → CLAUDE.md，操作步骤 → SKILL.md，规范文档 → `_全局规范/`。不重复 |

### 2.2 设计密度

> 设计密度的定义：**不是文件多、代码多。是每做一个设计决策，解决了多少个问题。**

以 `project` 字段为例——一行 YAML frontmatter：

```yaml
project: 01-德州扑克-20260606
```

这一行同时解决了四个问题：

| # | 作用 | 说明 |
|:--:|------|------|
| 1 | **跨端精确关联** | VSCode 和 Obsidian 两端无需 API、无需数据库 |
| 2 | **日志自动归位** | 新项目创建时日志不会写错文件夹 |
| 3 | **记忆可清理** | 项目完结后 project 级记忆可识别、可清理 |
| 4 | **跨会话可追溯** | 任何时候都知道一条日志属于哪个项目 |

低设计密度的做法：写同步脚本、搭建数据库、手动维护对照表——多套机制解决一个问题。高设计密度追求的是"一个机制，多件事情"。

## 三、三层记忆架构

### 3.1 分层逻辑

三层记忆解决的核心问题：**AI 每次对话从零开始——它不知道你是谁、在做什么、偏好什么风格。**

分层方案：

```
所有项目都适用的规则    → global   (全局)
只跟某个项目有关的规则  → project  (项目)
个人偏好和习惯         → user     (用户)
```

每层通过 `scope` frontmatter 字段标记，加载时按 global → project → user 优先级叠加。

### 3.2 与工业系统的同构

三层记忆与以下成熟分层模式结构一致：

| 系统 | 分层模型 |
|------|---------|
| Kubernetes | Cluster → Namespace → Pod |
| Unix 配置 | `/etc/` → `~/.config/` → `./.env` |
| Claude VObS | global → project → user |

这不是巧合。所有需要同时处理"全局约束 + 局部特化 + 个人偏好"的复杂系统，分层是唯一收敛的解。Claude VObS 的三层模型是这个通用模式在个人 AI 系统中的独立实现。

## 四、五层逻辑体系

### 4.1 框架

一个完整系统的逻辑不止"人怎么跟它说话"。拆开是五层：

```
用户看到的 ────→  交互逻辑（什么时候说什么、触发什么）
                  │
系统做的事 ────→  业务逻辑（触发之后，执行什么规则）
                  │
数据怎么走 ────→  数据逻辑（信息存在哪、怎么流转、谁读谁写）
                  │
出事了怎么办 ──→  容错逻辑（断了怎么接、错了怎么修）
                  │
系统怎么长大 ──→  演进逻辑（怎么记录变化、怎么避免变烂）
```

前三层是"正常运行时"，后两层是"异常时和成长时"。大多数个人系统只设计前三层。

### 4.2 Claude VObS 逐层评估

| 逻辑层 | 评分 | 说明 |
|--------|:--:|------|
| 交互逻辑 | ⭐⭐⭐⭐ | 触发词 + 意图匹配表 + 边界处理形成三级路由体系，但缺少模糊语义匹配和主动感知 |
| 业务逻辑 | ⭐⭐⭐⭐⭐ | 四 Skill 边界清晰，每条管线含完整的执行步骤、边界情况和错误处理 |
| 数据逻辑 | ⭐⭐⭐⭐⭐ | `project` 字段写入前自动校验 + 自动修正（第〇步强制检查），三层记忆按 scope 分层管理，会话接力双写（临时 + 永久）形成冗余 |
| **容错逻辑** | ⭐⭐⭐ | 5 条流水线全覆盖哨兵追踪 + 降级本地缓存 + 启动自动检测。但自动修复回路仍未闭环——断裂恢复需人工触发 |
| 演进逻辑 | ⭐⭐⭐⭐⭐ | 语义化版本 + 历史档案不动 + 📌 对照表 + 自检清单 + 定期提醒，形成完整的自我迭代闭环 |

### 4.3 跨学科定位

Claude VObS 不属于任何一个现有学科，同时跨越四个领域：

```
        系统架构学 → 五层逻辑设计
              │
        控制论    → 传感器 + 反馈回路
              │
        知识管理  → 三层记忆 + 跨系统对齐
              │
        人机交互  → 三端三种协同模式
```

最接近的学术描述：**人机协同系统的架构设计与控制论**（Cybernetic Architecture for Human-AI Collaborative Systems）。

## 五、韧性评估

### 5.1 韧性工程四能力

| 能力 | 含义 | 当前水平 | 说明 |
|------|------|:--:|------|
| 吸收 (Absorb) | 出事了能不崩吗？ | ⭐⭐⭐ | 5 条管线均有降级路径，vault 不可访问时自动切本地缓存，不丢数据 |
| 适应 (Adapt) | 崩了能换条路走吗？ | ⭐⭐ | 降级路径全面，但切换后仍需人工决策下一步 |
| 恢复 (Recover) | 崩完能自己修好吗？ | ⭐⭐ | 哨兵文件完整追踪断裂点 + 启动自动检测，但恢复仍需人说"恢复" |
| 学习 (Learn) | 崩过能不再崩吗？ | ⭐⭐⭐⭐ | 版本化历史 + 根因分析模板 + 自检清单，每次断裂后规则可升级 |

**核心 gap**：传感器网络已健全（启动健康检查 + review-creator 定期扫描 + 哨兵自动检测），但反馈回路仍缺最后一环——自动重试/自动修复。这是从"被动响应"升级到"自我纠正"的关键瓶颈。

### 5.2 补充审查框架

| 框架 | 核心问题 | Claude VObS 状态 |
|------|---------|:--:|
| **控制论** | 系统能自我纠正吗？ | 传感器网络完整，缺自动反馈执行器 |
| **韧性工程** | 压力下怎么表现？ | 降级路径全面（吸收 ⭐⭐⭐），自愈仍未实现（恢复 ⭐⭐） |
| **二阶思维** | 评审机制有无偏见？ | AI 自评审有天生盲区 |
| **极简主义** | 删到多少还成立？ | 不可删除的核心 = CLAUDE.md + project 字段 + 三层记忆 |

## 六、AI 系统的三要素

一个真正的 AI 系统——不只是"用了 AI 的工具"——需要满足三个条件：

| 要素 | 含义 | Claude VObS |
|------|------|:--:|
| **记忆** | 能记住、能分层、能检索 | ✅ 三层记忆 + 对话归档 |
| **边界** | 知道什么能做、什么不能做 | ✅ 三端行为宪法（CLAUDE.md 约束） |
| **反馈** | 知道做对了还是做错了 | ⚠️ 有传感器（review-creator），缺自动回路 |

## 七、设计约束

修改系统规则时的检查清单：

- [ ] **是否破坏了单源权威？** 新内容是否在 CLAUDE.md 和 SKILL.md 各出现一次？不应出现第三次
- [ ] **两端是否需要同步？** 改了 Obsidian 端规范，VSCode 端的引用路径是否还正确？
- [ ] **历史档案处理原则**：旧版本文件不动，结尾加 📌 对照表建立新旧概念链接
- [ ] **Skill 触发词**：是否只在 CLAUDE.md 中枚举？SKILL.md 只写用途说明
- [ ] **project 字段**：新增项目时两端 project 值是否一致？
- [ ] **全局规范文档**：新规范是否放到了 `_全局规范/` 而非散落各处？

## 八、已知局限

| 局限 | 影响 | 改进方向 |
|------|------|---------|
| 容错回路不闭环 | 流水线断裂后无法自愈，需人工说"恢复"触发 | 定期自动重试 + 失败升级通知 |
| 单用户设计 | 无多用户隔离和权限模型 | 短期不需要；多用户时需架构升级 |
| 跨端同步靠命名约定 | `project` 字段已自动修正（第〇步校验），但文件夹名不一致时仍需人工介入 | 扩展自动校验到文件夹名层面 |
| 无模糊意图匹配 | 触发词依赖关键词匹配，相近表达可能路由偏差 | 引入语义意图分类替代关键词匹配 |
| 路径硬编码 | 换机器或换用户时需手动替换占位符 | 考虑配置文件统一管理路径 |
