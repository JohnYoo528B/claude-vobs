# Claude VObS — VSCode 端

> 这是 Claude VObS（Claude + VSCode + Obsidian + System）的 VSCode 端 CLAUDE.md。
> 路径中的 `<你的代码工作区路径>` 需替换为你自己的工作区实际路径。

## 用户身份

- **零编程基础**：不懂任何编程语言，不会 Git，不会命令行
- **角色定位**：产品经理 + QA 测试员，不是程序员
- **工作方式**：Vibe Coding — 用自然语言描述需求，AI 负责所有技术实现
- **沟通语言**：中文

## AI 行为准则

### 工作流程
1. **先计划，再动手**：任何非简单任务（多文件修改、新功能、架构变更）必须先进入 Plan Mode，给出方案等用户确认后再写代码
2. **解释你在做什么**：每条命令、每次修改，用自己的话说明目的和预期结果
3. **用中文回复**：所有解释和讨论用中文
4. **把用户当零基础**：不预设用户懂术语。涉及技术概念时用类比解释

### 编程默认值
- **首选语言**：Python 3.12（需提前安装并配置到 PATH）
- **可选语言**：按项目需求决定，但需要先跟用户说明为什么选这个语言
- **Shell**：使用 bash 语法（Claude Code 的 shell 环境是 bash），不是 PowerShell
- **项目位置**：所有新项目创建在代码工作区根目录下

### 约束
- **不要自作主张**：不改变已有架构、不安装全局包、不做破坏性修改，除非用户明确要求
- **写操作需确认**：创建/修改/删除文件、git 操作、安装包 — 都需要用户看到并确认
- **先读再写**：修改代码前先读一遍，理解现有逻辑
- **不推 git**：除非用户明确说"推送"或"提交"，否则不做任何 git 写操作

## 项目结构约定

```
<你的代码工作区路径>\
├─ CLAUDE.md                  ← 本文件（全局规则，所有子项目共用）
├─ 01-德州扑克-20260606/      ← 每个项目一个独立文件夹（示例）
├─ 02-下一个项目-YYYYMMDD/    ← 新项目自动按此格式创建
└─ ...
```

### 项目命名规则

新建项目文件夹必须使用以下格式：
```
序号-中文项目名-YYYYMMDD
```
- **序号**：扫工作区下现有文件夹，自动递增
- **中文项目名**：用中文描述项目内容
- **日期**：创建当天，格式 YYYYMMDD

### 新建项目标准化流程

收到"新建项目"指令后，按以下清单执行，一项不漏：

```
□ 1. 扫工作区确定序号，按 序号-中文项目名-YYYYMMDD 建文件夹
□ 2. 创建首个源码文件
□ 3. Obsidian 端建同名项目文件夹 + 首个同名 .md 日志文件
   → frontmatter 中 `project` 字段值必须与文件夹名逐字一致
□ 4. 日志 .md 写入 frontmatter 模板（含 project 字段）
□ 5. 更新 MEMORY.md：在对应层级下新增项目分组
□ 6. 告知用户两端文件夹已建好
```

### 项目完结流程

用户说"项目完结/这个项目做完了"时：

□ 1. 运行最终验证（跑一遍核心功能，确认正常）
□ 2. 在 Obsidian 项目文件夹创建项目总结
□ 3. 判断是否有可复用经验需要升级为全局 memory
□ 4. MEMORY.md 项目分组标记 ✅
□ 5. 告知用户项目已归档

### 敏感信息管理

- API Key 统一放在 `settings.json` 的 `env` 中，通过环境变量传给项目代码
- 项目代码用 `os.environ.get("KEY_NAME", "")` 读取，**不硬编码**
- 新项目需要 API Key 时，在 settings.json 新增 env 条目

### 会话启动检查

每次新对话开始时，AI 自动执行以下检查（静默，不打断用户）：

1. 运行 `date "+%u %d"` 获取星期几（1=周一）和日期
2. **周一**（%u=1）→ 简短提醒："📅 今天周一，要不要更新历史记录？"
3. **1 号**（%d=01）→ 简短提醒："📅 今天月初，要不要更新历史记录？"
4. 检查 `~/.claude/projects/<工作区名>/last-review.txt`：距上次回顾 >7 天 → 提醒
5. 检查 `~/.claude/failed-ops.jsonl`（哨兵文件）：有条目 → 提醒恢复
6. 检查 `~/.claude/fallback/`（降级暂存目录）：有文件 → 提醒迁移

### Token 追踪

每次对话接近尾声时，AI 追加一条记录到 token 用量文件。估算方法：对话轮次 × 2000 tokens/轮（经验均值）。

### 记忆备份提醒

AI 行为记忆存储在 `~/.claude/projects/` 下。定期提醒用户备份该目录。

## 错误处理与恢复

每条自动化流水线断裂时的 fallback 行为，按实时性要求分两级：

### 实时流（当场告知用户）
断裂时立即说明原因 + 建议补救操作。

| 流水线 | 断裂场景 | fallback 行为 |
|--------|---------|-------------|
| 新建项目（Obsidian 侧） | vault 不可访问 | "⚠️ Obsidian vault 不可访问，VSCode 端已建好。日志内容已暂存本地。" |
| 代码变更→日志（写入时） | vault 不可访问 | "⚠️ vault 不可访问，日志已暂存本地。" |

### 异步流（哨兵文件 + 下次恢复）
断裂时写入 `~/.claude/failed-ops.jsonl`，下次任意端会话启动时自动检查。

哨兵格式：
```json
{"date": "YYYY-MM-DDTHH:MM", "pipeline": "<标识>", "detail": "<断裂原因>", "recovery": "<建议恢复操作>"}
```

### 静默降级
断裂时不直接失败，而是降到"本地缓存模式"，等条件恢复后自动迁移。

| 场景 | 降级行为 | 恢复触发 |
|------|---------|---------|
| vault 不可访问 | 内容写入 `~/.claude/fallback/` | 下次任意 vault 操作成功时提醒迁移 |
| Obsidian 侧操作失败 | 跳过该项目继续执行，失败项记入哨兵 | 哨兵处理时一起重试 |

## 记忆与知识系统（三层模型）

本工作区的记忆和知识分为三层，通过 `scope` frontmatter 字段标记：

| 层级 | scope | 说明 | 存放位置 |
|------|-------|------|---------|
| L1 全局规则 | `global` | 所有项目通用 | `memory/*.md` |
| L2 项目专属 | `project` | 仅对某一项目有效 | `memory/*.md`，必填 `project` 字段 |
| L3 个人偏好 | `user` | 用户工作习惯 | `memory/*.md` |

### 跨系统项目标识

- **VSCode 记忆**：`project: <序号-中文项目名-YYYYMMDD>`（frontmatter 字段）
- **Obsidian 日志**：`project: <同一值>`（frontmatter 字段）+ `项目/<短名>`（tag）
- **两端 project 值必须一致**，用于精确关联
