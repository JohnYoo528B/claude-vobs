# 量化判断标准

> **收工（PATH_C）判定引擎的唯一权威来源**。把原来 AI 的「自问」替换为可执行、可验证的判定规则。
>
> 使用方：review-dispatcher PATH_C 判定引擎 · review-creator Phase 4（原理判定）

---

## 一、问题一：「这次改动是否触及系统设计原理？」

### 1.1 致命条件（任一命中 → 降级 PATH_A 完整更新）

这些条件不需要判断——命中即触发完整更新：

| # | 致命条件 | 检查方法 | 来源 |
|:--:|---------|---------|------|
| F1 | 修改了 `系统设计原理.md` | `git diff --name-only` 包含该文件 | 原理门禁 |
| F2 | 修改了 `跨端共享规范.md` | `git diff --name-only` 包含该文件 | 跨端规则唯一权威 |
| F3 | 新增或删除了 skill | `ls ~/.claude/skills/` 前后对比，目录数量变化 | 管道层变更 |
| F4 | 改变了数据流向（新增/删除跨端数据管道文件） | 检查 `arch-changes.jsonl` 等数据管道文件是否被新增/修改 | 数据逻辑层变更 |

### 1.2 高权重条件（≥2 个命中 → 降级 PATH_A，仅 1 个 → 问用户）

| # | 高权重条件 | 检查方法 | 来源 |
|:--:|-----------|---------|------|
| H1 | 新增/删除/重命名了文件夹（非临时备份文件夹） | `git diff --stat` 含 `create mode` 或 `delete mode` 的行 | 内容架构变更 |
| H2 | 修改了 CLAUDE.md 意图路由表（路由行变化） | `git diff CLAUDE.md`，检查路由表 `| # | 意图 | ... |` 区域是否有增删改 | 交互逻辑层变更 |

### 1.3 中权重条件（不影响判定，仅记录在健康信号中）

| # | 中权重条件 | 检查方法 | 来源 |
|:--:|-----------|---------|------|
| M1 | 新增了系统性机制（如新检测项、新管道步骤） | 检查是否创建了新的规范文件（非笔记内容） | 演进逻辑层 |
| M2 | 文件被重命名（非文件夹） | `git diff --stat` 含 rename 行 | 可能影响引用 |

### 1.4 判定流程

```
git diff --stat
    │
    ├─ 命中任一致命条件（F1-F4）
    │   └─ PATH_A 完整更新（不由分说，通知用户即可）
    │
    ├─ 命中 ≥2 个高权重条件（H1-H2）
    │   └─ PATH_A 完整更新（告知用户理由）
    │
    ├─ 命中 1 个高权重条件
    │   └─ 问用户：「只有 <条件描述>，建议走微调。但如果你觉得改动够大就做完整更新。微调还是更新？」
    │
    └─ 全部未命中
        └─ 进入问题二（判定大改/中改/小修）
```

## 二、问题二：「大改还是小修？」

### 2.1 量级判定规则

| 量级 | 判定标准 | 映射路径 | 例子 |
|:--:|---------|:--:|------|
| **大改** | 改动文件数 ≥ 3 **且** 涉及 ≥ 2 个规范文件（`设计规范/核心设计规范/` 下文件 + CLAUDE.md + Skill SKILL.md） | PATH_A（建议完整更新） | 新建 2 个规范文件 + 改 review-creator SKILL.md + 改 CLAUDE.md |
| **中改** | 改动 1-2 个规范/架构文件（规范文件 + 路由表 + Skill 文件） | PATH_B（微调） | 给 system-watch 加第 6 项 |
| **小修** | 仅文案修正 / 格式调整 / 单文件非规范文件追加（非规范文件 = 笔记内容 .md） | PATH_B（微调） | 修一个错别字、补一句说明 |
| **跳过** | `git diff --stat` 无输出，且无 arch-changes.jsonl 未消费条目 | 跳过更新，仅记忆 | 只聊天没改任何文件 |

### 2.2 辅助判断指标

以下指标帮助确认量级，不是硬性规则：

| 指标 | 大改倾向 | 小修倾向 |
|------|:--:|:--:|
| 有断链修复需求 | ✅ | — |
| 涉及使用指南五节中任意节更新 | ✅ | — |
| 改动跨越 vault + VSCode 两端 | ✅ | — |
| arch-changes.jsonl 有未消费条目 | ✅ | — |
| 仅修改单文件中的文字表述 | — | ✅ |
| 仅追加不修改现有内容 | — | ✅ |

### 2.3 判定流程图

```
改动文件列表 + 规范文件判定
    │
    ├─ 文件数 ≥ 3 且 规范文件 ≥ 2
    │   └─ "大改" → 建议 PATH_A，告知用户理由
    │
    ├─ 1-2 个规范/架构文件
    │   └─ "中改" → PATH_B 微调
    │
    ├─ 仅文案/格式/笔记内容文件
    │   └─ "小修" → PATH_B 微调
    │
    ├─ git diff 无输出
    │   └─ "跳过" → 仅记忆，不更新历史记录
    │
    └─ 拿不准
        └─ 默认 PATH_B（微调），并在健康信号中标注「AI 判定拿不准，建议用户复核」
```

## 三、收工完整判定决策树（合并问题一+问题二）

```
用户说"收工/结束"
    │
    ├─ Step 1: git diff --stat 取改动文件列表
    │     │
    │     ├─ 无改动 + 无 arch-changes 未消费项
    │     │   └─ 跳过更新 → 仅记忆（#16 #17 #18）
    │     │
    │     └─ 有改动 → Step 2
    │
    ├─ Step 2: 检查致命条件（F1-F4）
    │     │
    │     ├─ 命中 → 🚨 PATH_A 完整更新 + 记忆
    │     │
    │     └─ 未命中 → Step 3
    │
    ├─ Step 3: 检查高权重条件（H1-H2）
    │     │
    │     ├─ ≥2 命中 → PATH_A 完整更新 + 记忆
    │     ├─ 1 命中 → 问用户选 PATH_A 还是 PATH_B
    │     └─ 0 命中 → Step 4
    │
    └─ Step 4: 量级判定（§二规则）
          │
          ├─ 大改 → 建议 PATH_A + 记忆
          ├─ 中改/小修 → PATH_B 微调 + 记忆
          └─ 拿不准 → PATH_B + 记忆（plus 标注）
```

## 四、使用规范

### 对 review-dispatcher

- 收工触发后，严格按 §三 决策树执行，不跳过步骤
- 问用户的场景（1 个高权重 + 拿不准量级），用 ≤2 句话说明理由
- 默认倾向：拿不准时偏向微调（PATH_B），不阻塞用户收工

### 对 review-creator Phase 4

- `principle_check="deep"` → 走完整原理门禁（含旧 §五 全部 6 条判定 + 原理版本化流程）
- `principle_check="light"` → 仅检查致命条件 F1-F4，不触发原理版本化

### 边界情况

- 用户强制要求：用户说「不管怎样走微调」→ 尊重用户，PATH_B
- 会话很短（< 3 轮无文件改动）→ 跳过更新，仅记忆
- 改动全是 AI 自己为修复路径做的 → 仍按规则判定（不因「AI 改的」降级）
