# 跨端共享规范

> 本文档是 Obsidian ↔ VSCode 跨端规则的**唯一权威来源**。两端 CLAUDE.md 各自引用本文，不重复维护同一规则。修改跨端规则时，只改这一个文件。系统当前版本见 [[_系统管理/系统健康/VERSION]]。

---

## 一、三端分工速览

| 端点 | 角色 | AI 定位 | CLAUDE.md |
|------|------|--------|-----------|
| **VSCode** | 工程师 | 写代码、跑数据、管理 git | `<你的 VSCode 代码区路径>\CLAUDE.md` |
| **Obsidian** | 编辑 | 管笔记、做归档、写评审 | 本 vault 根目录 `CLAUDE.md` |
| **手机** | 秘书 | 快速捕捉、搜索 vault | 通过 cc-connect 路由到 skill |

手机↔电脑 的问题本质是**数据传输**（消息到达 → 写入 vault），已通过 quick-capture + fallback 解决，见各 Skill 的 SKILL.md。

本文档聚焦 **Obsidian↔VSCode**，即两台「有 AI 的终端」之间的规则一致性。

## 二、GitHub 公开版发布

### 触发词

| 你说的 | AI 做什么 |
|--------|----------|
| "发布 GitHub 新版本 / 发布到 GitHub / GitHub 发布 / 上线 / 发布公开版" | **完整发布流程**（本节 ⬇） |
| "推送到 GitHub" | 反问确认：是要完整发布（tag + release），还是只 git push？ |

### 发布流程（Obsidian 端 AI 执行）

1. **确定版本号**：从 `_系统管理/系统历史记录/` 最新版本文件名提取。用户指定了版本号则用用户的。
2. **呈现确认**：列出变更摘要 + 版本号，等用户确认。
3. **写上线日志**：创建 `_系统管理/GitHub公开版/v<版本号>-上线日志.md`，全文档时间戳精确到分钟。
4. **更新总览**：更新 `_系统管理/GitHub公开版/GitHub公开版-总览.md`。
5. **VSCode 端同步**（Obsidian AI 跨端操作 VSCode 仓库）：
   - `cd e:/6_CodeData/00-Claude-VObS-公开版-20260608`
   - **同步 CLAUDE 文件**：`CLAUDE_Obsidian.md` ← vault `CLAUDE.md` · `CLAUDE_VSCode.md` ← `e:\6_CodeData\CLAUDE.md`
   - **同步 Skills**：复制 `~/.claude/skills/` 下所有 VObS 自定义 skill 到 `skills/`
   - **同步 DESIGN 文档**：`DESIGN.md` / `DESIGN_zh.md` ← 从 `系统设计原理.md` 映射更新
   - **同步 docs/**：`docs/跨端共享规范.md` ← vault 当前版本 · 检查其他 docs 文件是否需要更新
   - **同步 templates/**：检查模板文件是否与 vault 一致
   - **更新 README**：按 §三 关联提醒模型更新 `README.md` + `README_zh.md`（含版本 badge `vX.X` → `vY.Y`）
   - **验证清单**：逐项确认以下文件在仓库中存在且为最新版本：
     - [ ] `skills/` 下所有 skill 文件
     - [ ] `docs/跨端共享规范.md`
     - [ ] `CLAUDE_Obsidian.md` + `CLAUDE_VSCode.md`
     - [ ] `DESIGN.md` + `DESIGN_zh.md`
     - [ ] `templates/` 下所有模板
     - [ ] 四层文件夹（`～草稿箱/`、`1主题研究/`、`2日志与笔记/`、`3工作与产出/`）及其 README.md
   - `git add -A && git commit -m "发布 v<版本号>: <简要描述>"`
   - `git tag -a v<版本号> -m "v<版本号>: <描述>"`
   - `git push origin master && git push origin v<版本号>`
   - `gh release create v<版本号> --title "v<版本号> — <标题>" --notes "<release notes>"`
6. **确认回复**：发布完成后，回复 GitHub Release 链接 + 上线日志路径。

### 版本号规则

- 大版本（如 4.0、5.0）→ 完整发布流程（tag + Release + 上线日志）
- 小版本（如 4.1、4.2）→ 仅改文件 + commit + push，不打 tag 不发 Release，不写上线日志
- 版本号取系统版本号（系统 4.0 → 发布 v4.0）
- **上线提醒阈值**：每 0.5 版本 AI 会自动提醒发布（如 v3.5 → v4.0 → v4.5 → v5.0），中间版本仅 commit+push，不打 tag 不发 Release。**此规则约束的是 AI——别再小版本就催用户发布。用户不受此限：用户说「发布」就发布，不论版本差多少。** 此规则为永久性设计决策，不受单次评审影响。

### 边界情况

- vault 不可访问：写哨兵 `pipeline: "github-release"`，告知用户
- gh CLI 未认证：告知用户检查 `gh auth status`
- tag 已存在：询问用户是否覆盖
- VSCode 仓库路径不可达：告知用户检查路径，暂停

### 数据隔离

`_系统管理/GitHub公开版/` 是 vault 的只读快照管理区。同步方向只有 vault → GitHub，**禁止 GitHub → vault**。

## 三、README 编写规范

### 核心原则：README 是独立原创文档，内部文档变了提醒检查对应段落

README 面向外部访客，用的是叙事语气、个人视角，与 vault 内部给 AI 读的执行规则是**两套表达体系**。内容改了任何一边，另一边跟着可能也要改——但不能机械提取，因为语气和结构完全不同。

**关联提醒模型**：README 的 15 段与内部文档建立关联映射。内部文档变更时 → 对照映射表 → 通读 README 对应段 → 判断叙事是否过时 → 过时则列出差异，让用户决定是否更新。

### 内容映射表：15 段 × 三种分类

#### 🟢 自动更新（发布时必然改动，不改叙事）

| README 节 | 来源 | 更新方式 |
|-----------|------|---------|
| §1 Badge 版本号 | [[_系统管理/系统健康/VERSION]] | 替换版本号 |
| §2 本版本更新 | [[_系统管理/系统历史记录/]] 最新版本 | 提取变更摘要，不改叙事语气 |

#### 🟡 关联提醒（内部文档变了 → 对照检查 → 列差异让用户决定）

| README 节 | 关联内部文档 | 检查要点 |
|-----------|-------------|---------|
| §3 我为什么做这个 | [[_系统管理/设计规范/核心设计规范/系统设计原理]] §一「系统本质」 | 系统定位、设计动机是否变化 |
| §4 我每天怎么用 | [[CLAUDE]] 路由表 + Skill 体系 | 触发词、工作流是否新增/变化 |
| §5 这是什么？ | [[_系统管理/设计规范/核心设计规范/系统设计原理]] §一 §二 | 三端分工、核心能力概述是否过时 |
| §6.1 三层内容架构 | [[_系统管理/设计规范/核心设计规范/系统设计原理]] §三 | 文件夹结构描述是否一致 |
| §6.2 MOC 机制 | [[1主题研究/README]] | MOC 列表、机制描述是否同步 |
| §6.3 Skill 路由系统 | [[CLAUDE]] + [[_系统管理/设计规范/核心设计规范/系统设计原理]] §五.2 | 触发词表、路由逻辑是否变化 |
| §6.4 跨端协作 | [[跨端共享规范]] §四 | 三端分工描述是否过时 |
| §6.5 三层记忆 | [[_系统管理/设计规范/核心设计规范/系统设计原理]] §二 + §五.3 | 记忆层机制是否变化 |
| §6.6 容错机制 | [[_系统管理/设计规范/核心设计规范/系统设计原理]] §五.4 | 哨兵/降级/断点机制是否变化 |
| §12 核心设计 | [[_系统管理/设计规范/核心设计规范/系统设计原理]]（整体） | 设计原则列表是否需增删 |

#### 🔵 README 独有（手动维护，不映射）

| README 节 | 内容 |
|-----------|------|
| §§7-8 前置条件 | Claude Code / VSCode / Obsidian / Python / Git 安装要求 |
| §9 快速开始 | 安装步骤（自动 + 手动），内部用户已装好，不需要内部文档 |
| §10 手机端 | 手机端使用方式，vault 内部不写 |
| §11 仓库结构 | 文件树图，每次改文件同步更新 |
| §13 FAQ | 常见问题 |
| §14 适合谁 | 目标用户描述 |
| §15 灵感 | 致谢/灵感来源 |
| §16 许可证 | MIT 声明 |

### 生成流程（每次发布时执行）

1. **自动更新**：badge 版本号 + §2 本版本更新（提取变更摘要，不改叙事语气）
2. **对照检查**：按 🟡 映射表，逐项确认关联内部文档本次是否改了 → 改了则通读 README 对应段 → 判断叙事是否过时 → 过时列出差异让用户决定是否更新
3. **独有内容**：扫 §§7-16，有变化才改
4. **英文版同步**：镜像结构更新，不逐字翻译

### 格式规则（继承）

- **读者假设**：写给第一次看见仓库的陌生人。每一步交代"在哪做、怎么做"，不预设前置知识
- **表格三分法**：多列对比 → HTML `<table>`；内容信息 → `-` 列表；简单两列 → MD 表格
- **中主英副**：中文 README 是主力。英文版镜像结构，两边内容对等。先改中文版再同步英文版
- **仓库结构是锚点**：改任何文件就同步改仓库结构图——它是唯一权威索引
- **定位优于功能列表**："为什么不用现成的"对比表 > 功能描述——10 秒内决定继续读还是关掉

## 五、跨端路径权威

每类路径有且仅有一个权威来源：

| 路径类型 | 权威来源 | 说明 |
|---------|---------|------|
| VSCode 代码区 (`e:\6_CodeData\`) | VSCode CLAUDE.md | 代码、git、公开版仓库 |
| Obsidian vault | Obsidian CLAUDE.md | 笔记、日志、系统管理 |
| `~/.claude/` (skills, memory, projects) | 两端共享 | 文件系统层面同一目录 |

**规则**：每条路径只在一端定义。需要引用对端路径时，通过 CLAUDE.md 中的"关联系统"节链过去，不复制粘贴。

### 路径对齐校验（每次启动执行）

两端 AI 在启动时执行以下具体检查（不是"看看对不对"，是逐条跑命令）：

**Obsidian 端检查：**
1. `grep "e:\\\\6_CodeData" e:\\6_CodeData\\CLAUDE.md` — 提取 VSCode CLAUDE.md 中引用的所有 vault 内部路径
2. 对每条路径，用 `ls <路径>` 验证存在性
3. 任一断裂 → `"⚠️ 跨端路径不匹配：<具体差异>，说「修复路径对齐」"`

**VSCode 端检查：**
1. 读 Obsidian CLAUDE.md 的 §笔记库结构，提取 vault 根路径
2. 用 `ls <vault根>/_系统管理/设计规范/` 验证关键子目录存在性
3. 将本端 CLAUDE.md 中引用的 vault 内部路径与实测对比
4. 任一断裂 → 以 Obsidian 端实测为准自动修正（不等待用户确认）

修复方向：路径以各端 CLAUDE.md 为权威来源。VSCode 端引用 vault 路径断裂时以 Obsidian 端实测为准；Obsidian 端引用 VSCode 路径断裂时提醒用户确认。

## 六、跨端触发词边界

| 触发词 | Obsidian 端含义 | VSCode 端含义 | 歧义处理 |
|--------|:--|:--|------|
| 「发布」/「上线」/「发布到 GitHub」 | 完整发布流程（tag + release） | 未定义 | Obsidian 端执行 |
| 「推送」/「推送到 GitHub」 | 反问确认后执行 | `git push`（无 release） | **先确认**是发布还是只 push |
| 「整理」/「归档」 | 对话→笔记存档 | 未定义 | — |
| 「记忆一下」 | 会话接力（双写） | 未定义 | — |
| 「提交」 | 引导用户去 VSCode 端操作 | 四步自动化（确认 commit→push→日志→README） | VSCode 端触发 |
| 「更新日志」 | 不用（只接收 VSCode 推送） | 代码→Obsidian 日志 | VSCode 端触发 |
| 「新建项目」 | 收到后配合建日志文件夹 | 6 步流程（含跨端步骤） | VSCode 端触发 |
| 「项目完结」 | 创建 回顾-项目总结.md | 5 步完结流程 | VSCode 端触发 |
| 「完结后维护」 | 追加"完结后维护记录"表格 | 修代码 → 补维护记录 | VSCode 端触发 |

**规则**：新增触发词时，先判断是否跨端。跨端 → 写入本文档。单端 → 写入对应 CLAUDE.md。

## 七、项目生命周期

### 阶段一览

| 阶段 | 触发词 | VSCode 端动作 | Obsidian 端动作 | 详细流程 |
|------|--------|-------------|----------------|:--:|
| 新建 | 「新建项目」 | 6 步流程 | 建日志文件夹 + 首个 .md | 见 §八 |
| 日常开发 | 「更新日志」 | dev-log-creator | 接收写入 | dev-log-creator SKILL.md |
| 完结 | 「项目完结」 | 5 步流程 | 创建 回顾-项目总结.md | 本节 ⬇ |
| 完结后维护 | （自然语言） | 修代码 → 补维护记录 | 追加"完结后维护记录"表格 | 本节 ⬇ |

### 完结流程

VSCode 端 AI 执行 5 步：
1. 运行最终验证（跑一遍核心功能，确认正常）
2. **跨端**：在 Obsidian 项目文件夹创建 `回顾-项目总结.md`（成果/收获/可复用资产/已知限制）
3. 判断是否有可复用经验需要升级为全局 memory（scope: global）
4. MEMORY.md 项目分组标记 ✅
5. 告知用户项目已归档

### 完结后维护规则

项目完结后的小修（Bug 修复、兼容性调整等，不涉及新功能或架构变更）：
- 在项目总结末尾追加 **"完结后维护记录"** 表格（日期 + 内容）
- 不改变完结状态，不走完结流程
- 如果是新功能（非小修），说明项目已重新激活，移除 ✅ 标记

### 架构变更记录（供回顾消费）

VSCode 端每次完成架构改动后，自动追加一行到 `~/.claude/projects/e--6-CodeData/arch-changes.jsonl`：

```json
{"date":"ISO时间","summary":"一句话标题","files":["文件1"],"before":"变更前","after":"变更后","impact":"为什么重要","consumed":false}
```

| 字段 | 作用 |
|------|------|
| `date` | 改动时间 |
| `summary` | 一句话，直接进入 review「架构变更」表格 |
| `files` | 涉及文件列表 |
| `before` | 变更前状态 |
| `after` | 变更后状态 |
| `impact` | 对系统设计的影响（供 review 第八步判定是否触发原理版本化） |
| `consumed` | 是否已被 review 消费（防重复） |

**触发写入的改动类型**：CLAUDE.md 规则变更、跨端共享规范修改、系统设计原理修改、memory 层架构调整、Skill 体系变更、文件夹重组。纯代码级变更走 dev-log-creator，不写此文件。

**Obsidian 端消费**：review-creator Phase 3（第七步）读取未消费条目，直接填入「架构变更」表。review 完成后标记 `consumed: true`。

> ⚠️ **关键修正（v4.3）**：所有回顾路径（完整更新 PATH_A / 微调 PATH_B / 收工 PATH_C）都标记 `consumed: true`。之前仅完整更新标记，导致微调和收工路径下 arch-changes 累积不清理。详细规则见 [[下游传播表]] #7。

## 八、新建项目跨端流程

> 详细步骤见 VSCode CLAUDE.md "新建项目标准化流程"。此处仅说明跨端部分。

VSCode 端 AI 执行 6 步：
1. 建 VSCode 项目文件夹
2. 创建首个源码文件
3. **跨端**：在 Obsidian vault 的 `2日志与笔记/VibeCoding开发日志/` 下建同名文件夹 + 首个日志 .md（frontmatter 中 `project` 字段 = 文件夹名，写入后回读校验）
4. 写 frontmatter 模板
5. 更新 MEMORY.md
6. 告知用户两端已建好

两端文件夹名**必须一致**。详细规范见 [[备忘-项目规范与开发日志流程]]。

## 九、评估模式标记

评审报告可在 frontmatter 中声明 `eval_mode` 字段，影响评分维度：

| 模式 | 含义 | 行为 |
|------|------|------|
| `framework-first` | 系统在框架搭建阶段，内容填充不是当前重点 | 内容产出维度跟踪但不评分 |
| （不声明） | 默认全维度评估 | 所有维度参与评分 |

设置为 `framework-first` 时，评分表中内容产出维度标注"框架优先模式，不参与评分"，不拉低综合分。

## 十、错误处理：跨端部分

两端共享同一套错误处理机制（重试→哨兵→降级），但流水线各有不同：

| 流水线 | 触发端 | 断裂场景 | 恢复指令 |
|--------|:--:|------|------|
| 新建项目 Obsidian 侧 | VSCode | vault 不可访问 | "补建 Obsidian 日志" |
| 代码→日志同步 | VSCode | vault 不可访问 | "更新日志" |
| 历史记录 review | Obsidian | review-creator 步骤出错 | "更新历史记录" |
| GitHub 发布 | Obsidian | VSCode 仓库不可达或 gh 未认证 | "发布 GitHub 新版本" |

哨兵文件位置：`~/.claude/failed-ops.jsonl`  
降级暂存位置：`~/.claude/fallback/`

## 十一、修改记录
