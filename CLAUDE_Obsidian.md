## 用户身份

- **零编程基础**：不懂任何编程语言，不会 Git
- **角色定位**：知识管理者，不是程序员
- **工作方式**：用自然语言描述想要的操作，AI 负责执行
- **沟通语言**：中文

## AI 行为准则

### 这是笔记库，不是代码工作区
- **首要任务**：整理、搜索、关联、总结笔记内容
- **不要主动写代码**：除非用户明确要求在笔记里记录代码片段或开发思路
- **Markdown 优先**：所有输出用 Markdown 格式，方便 Obsidian 渲染和链接
- **善用 Obsidian 特性**：双向链接 `[[]]`、标签、frontmatter、Callout 等

### 沟通规则
- 消息中出现「明白吗 / 明不明白 / 是否明白」时：**视为最高优先中断。立即停止所有动作，先用自己的话解释理解了什么，等用户确认「对」后再执行。任何情况下不得跳过。**
- **写时间戳前必须跑 `date` 命令**：会话记录、历史记录、文件 frontmatter 中的时间，一律以 `bash: date` 输出为准，不凭感觉写
- **改动后自检**：涉及系统架构、skill 变更、名称/文件夹变动时，改动完成后自动重新通读所有受影响的文件，检查是否引入路径断裂、逻辑矛盾、命名不一致等 bug，有则当场修复

### 工作流程
1. **搞清楚需求**：用户想整理什么、总结什么、查找什么
2. **并发搜索两处**：vault 笔记（Grep/Glob）+ 对话记忆（JSONL），两处结果同时呈现，让用户选要看哪个
   - 用户说"找/搜索/查一下/有没有关于 xxx"→ 必须同时搜 vault 和对话记忆
   - 呈现格式：列出两边的匹配结果，标注来源（📁 vault / 💬 对话记忆），让用户选
3. **执行**：整理、提炼、关联
4. **解释结果**：做了什么、为什么这样做

### 约束
- **不要修改已有笔记**：除非用户明确要求编辑某篇
- **新建笔记需确认位置**：告诉用户你打算把新笔记放在哪个文件夹
- **不要删除任何文件**：除非用户明确说"删除"
- **笔记风格保持一致**：注意现有笔记的格式和语气，不要突然变风格

### 手机端快捷操作（通过 cc-connect / 微信）

- **快速捕捉**：`存/暂存/记录/存一下 xxx` / `以上都存一下` / `以上都存下来` / `以上全存下来` → `quick-capture`，写入 `00草稿箱/手机暂存/`，按 session 分文件
- **搜索 Vault**：触发词 `找/搜索/查一下/有没有关于` → 路由到 `vault-search` skill

> 操作步骤见各 skill 的 SKILL.md（`~/.claude/skills/<skill名>/SKILL.md`），此处仅做路由。
> ⚠️ **手机端记忆拦截**：手机端（cc-connect/微信）收到 `记忆一下` / `保存记忆` 等记忆请求时，**不直接执行**，先回复确认句，等用户确认后再写。

- 核心词含"存/暂存/记"且不含"整理/归档/归整" → 走快速捕捉（原文照存）

判断优先级：
1. 先看信号词命中哪个意图
2. 命中多个时，选匹配词最多的
3. 实在拿不准 → 列出 2-3 个可能的意图让用户选

## 对话→笔记存档

用户说 `整理成笔记` / `整理一下` / `归档` 时触发。

### 执行流程
1. **回顾对话**：梳理核心问题、决定、洞察
2. **提炼 3-5 条结构化要点**（不逐字复述）
3. **判断归属文件夹**：从 `1学习笔记/2工作文档/3投资记录/4创作灵感/5其他知识/0VibeCoding开发日志` 中选择，拿不准时列出理由让用户选
4. **按模板生成预览**：frontmatter（date + type + status + tags）+ 正文 + `## 相关链接`
5. **搜索关联笔记**：用 Grep 搜索 vault 找相关笔记，填到相关链接节
6. **展示预览，等用户确认后再写入**

### 边界情况
- 对话超 20 轮：只提炼最核心 3-5 点，标注"精简版"
- 无实质内容：回复"这次对话主要是操作性的，没有需要存档的讨论"
- 话题跨领域：列出选项让用户选
- 已有类似笔记：提示"已有 [[xxx]]，要合并还是新建？"

## 草稿箱清理

用户说 `整理草稿箱` / `清理草稿箱` 时触发。

### 执行流程
1. **全面分析**：逐个检查 `00草稿箱/` 下所有文件（含 `手机暂存/` 子文件夹），判定每条归属
2. **呈现待确认计划**：列出每个文件的判定 + 将执行的操作，表格呈现
3. **确认后执行**：用户确认后逐文件操作

## 会话连续性

### 启动时
读完 CLAUDE.md 后：

**日期检测**（先执行，有提醒则暂停后续步骤）：
1. 运行 `date "+%u %d"` 获取星期几（1=周一）和日期
2. **周一**（%u=1）→ 提醒："📅 今天周一，要不要更新历史记录？"
3. **1 号**（%d=01）→ 提醒："📅 今天月初，要不要更新历史记录？"

**常规检查**：
1. 检查 `.claude/上次会话.md`，有则读取并在开场白中引用上次上下文
2. 检查 `~/.claude/failed-ops.jsonl`（哨兵文件），有条目则提醒用户
3. 检查 `~/.claude/fallback/`（降级暂存目录），有文件则提醒用户
4. **任务扫描**：Grep `status: (todo|doing)` 按 `layer` 分组，以简洁表格呈现待办摘要
5. **系统健康检查**（静默执行，仅在发现问题时提醒）

### 结束时（触发词：`记忆一下` / `把这段记忆下来` / `保存记忆`）
1. 1-3 句话总结本次会话做了什么
2. **扫描待办**：Grep `status: (todo|doing)` → 将结果填入「未完成」列表
3. 列出未完成事项
4. **双写**：
   - `.claude/上次会话.md`（覆盖写，快速接力）
   - `_系统管理/模型对话记忆/<YYYY-MM>.md`（同一月内追加，永久留存）
5. **确认回复**：写入完成后，回复一段确认文字

## 智能链接建议

### 自动触发（创作/编辑笔记后）
从笔记提取 2-5 个核心概念词，用 Grep 搜索 vault，筛选排序后提出建议。

匹配规则：
- **高匹配**（核心概念命中 ≥2 或不同文件夹但同领域）：**自动写入**
- **低匹配**（命中 1 个，概念关联较弱）：列出询问
- **同文件夹内**：不自动链接（文件列表已体现关系）

写入格式（统一，正文末尾，与正文明显区分）：
```markdown
---

> 📎 **相关笔记**：[[路径/笔记名]] · [[路径/笔记名]] · [[路径/笔记名]]
```

### 不触发的情况
- 模板/ 目录
- 内容 < 50 字
- type 是 `moc`
- 批量操作（同轮 >3 文件）
- 刚拒绝过同一篇的建议
- 连续两次被拒，当天不再主动建议

## 架构版本号规则

版本号只保留小数点后一位（`2.0` → `2.1` → `3.0`）。小修补内容直接合并到当前版本，不另开小版本号。

文件命名：`v<版本号>版本-<YYYYMMDDHHMM>.md`（精确到分钟，如 `v3.0版本-202606080255.md`）。

## 错误处理与恢复

每条自动化流水线断裂时的 fallback 行为，按实时性要求分两级：

### 实时流（当场告知用户）
断裂时立即说明原因 + 建议补救操作，不等下次会话。

| 流水线 | 断裂场景 | fallback 行为 |
|--------|---------|-------------|
| 移动端捕捉 | cc-connect 掉线，内容入本地暂存 | 回复"捕捉失败：cc-connect 连接异常。内容已暂存本地，下次 vault 正常时自动同步。" |
| 会话接力（写入失败） | `.claude/上次会话.md` 无写入权限 | 回复"会话记录写入失败（权限问题）。内容保留在对话中，下次可手动补记。" |
| 对话归档（单文件写入失败） | 归档时个别文件权限不足 | 跳过该文件，其余继续。失败文件记入哨兵 |

### 异步流（哨兵文件 + 下次恢复）
断裂时写入 `~/.claude/failed-ops.jsonl`（一行一条 JSON），下次任意端会话启动时检查并提醒用户。

哨兵格式：
```json
{"date": "YYYY-MM-DDTHH:MM", "pipeline": "<标识>", "detail": "<断裂原因>", "recovery": "<建议恢复操作>"}
```

| 流水线 | pipeline 标识 | 断裂场景 |
|--------|-------------|---------|
| 历史记录 | `review` | review-creator 某步命令出错 |
| 移动端捕捉 | `quick-capture` | cc-connect 掉线导致写入不完整 |
| 会话接力 | `session-checkpoint` | vault 不可访问，无法写模型对话记忆 |
| 开发日志 | `dev-log` | dev-log-creator 写入失败 |
| 对话归档 | `archive` | 归档写入时 vault 不可访问 |

### 静默降级
断裂时不直接失败，而是降到"本地缓存模式"，等条件恢复后自动迁移。

| 场景 | 降级行为 | 恢复触发 |
|------|---------|---------|
| vault 不可访问 | 内容写入 `~/.claude/fallback/` | 下次任意 vault 操作成功时提醒迁移 |
| 单文件写入失败 | 跳过该文件，其余继续，失败列表记入哨兵 | 哨兵处理时一起重试 |

### 启动检查
每次会话启动时，AI 静默执行以下检查：

**哨兵检查**（`~/.claude/failed-ops.jsonl`）：
- 无文件或无未处理条目 → 跳过
- 有条目 → 简短提醒

**降级恢复检查**（`~/.claude/fallback/`）：
- 目录为空或不存在 → 跳过
- 有文件 → 提醒迁移到 vault
