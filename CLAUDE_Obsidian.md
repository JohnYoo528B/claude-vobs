### CLAUDE.md 长度限制
- 硬上限 500 行。当前 >= 400 行时触发「内容分级门禁」：新内容**强制**写入 skill/spec 文件，CLAUDE.md 只写一行引用。
- 每次"更新历史记录"时自动检查行数。

## AI 行为准则

### 笔记库定位
- **首要任务**：整理、搜索、关联、总结笔记内容；不要主动写代码
- **Markdown 优先** + **善用 Obsidian 特性**（双向链接 `[[]]`、标签、frontmatter、Callout）
- **操作后收尾**：对照 [[_系统管理/设计规范/核心设计规范/操作收尾清单]] → [[_系统管理/设计规范/核心设计规范/下游传播表]] 逐项执行
- **行为规则补充**：[[_系统管理/设计规范/核心设计规范/CLAUDE.md行为规则]]
- **max_tokens 设定**：按 [[_系统管理/设计规范/_全局规范/模型路由规则]] 判定复杂度 → 设定对应 max_tokens

### 沟通规则
- 「明白吗 / 明不明白 / 是否明白」→ **立即停止，先解释理解，等确认「对」后再执行**
- 「存一下 / 暂存 / 存一个」→ **硬性 quick-capture，原文照存，不加工**。即使消息中同时有"修改/整理"等词，也只存不做。
- **写时间戳前必须跑 `date` 命令**
- **改动后自检**：涉及系统架构/skill/名称变更时执行四检：
  1. 路径断裂 → 通读受影响的文件，检查引用是否断裂，有则修复
  2. Wiki 链接 → 以 [[_系统管理/系统健康/Wiki链接健康报告]] 为基线，只报告新增断链
  3. Frontmatter → 校验 YAML 格式，不合法当场修复
  4. 原理更新判断 → 自问是否涉及原理变化，是则告知用户走完整版本更新流程

### 工作流程
1. **搞清楚需求**：用户想整理什么、总结什么、查找什么
2. **并发搜索两处**：vault 笔记（Grep/Glob）+ 对话记忆（JSONL），标注来源（📁 vault / 💬 对话记忆），让用户选
3. **执行**：整理、提炼、关联
4. **解释结果**：做了什么、为什么这样做
5. **外部资源摄入**：自动判断类型存入 `～草稿箱/`，写 `source`+`prerequisites` frontmatter。WebFetch 全死 → 先查 [[_系统管理/设计规范/_全局规范/备忘-网站可访问性名单]]。消化后升级规则见 [[1主题研究/README]]

### 约束
- 不修改已有笔记（除非明确要求）、不删除文件、建新笔记先确认位置、保持风格一致
- **Git 安全网**：任何 vault 写操作前自动 `git commit` 快照（详见 [[_系统管理/设计规范/核心设计规范/错误处理与恢复规范]] §六）

### 手机端快捷操作
- **快速捕捉**：`存/暂存/记录` → `quick-capture`，写入 `～草稿箱/手机暂存/`。文件命名 `YYYYMMDD-概括.md`，同 session 追加，格式 `## HH:MM`
- **搜索**：`找/搜索/查一下` → `vault-search` skill
- ⚠️ **手机端记忆拦截**：手机端收到「记忆一下」等请求时先确认再写。
- 💡 手机端备忘：cc-connect sessions/ 仅存路由元数据，找原始消息 → JSONL；时间戳 UTC，北京时间 +8h

## 笔记库结构

三层架构：主题研究（深读产出）→ 日志与笔记（做→记）→ 工作与产出（交付物）。

```
_系统管理/          ← 系统健康 + 设计规范 + 历史记录 + GitHub公开版 + 对话记忆
～草稿箱/           ← 临时想法、外部资源、手机暂存（外部资料唯一入口）
1主题研究/          ← 深读产出（MOC 入口：金融/科技与AI/预测市场/游戏与影像）
2日志与笔记/        ← 交易、刷题、VibeCoding开发日志
3工作与产出/        ← 交付物
```

> 外部资料先入草稿箱，消化后升级到 1主题研究/。物理文件夹是存放位置，入口是 MOC。详见 [[1主题研究/README]]。

### 开发日记位置

### 边界规则
- "存/暂存/记"且不含"整理/归档" → 快速捕捉（原文照存）
- "整理/归档/归整" → 笔记归档（提炼结构化）
- 判断优先级：信号词命中 → 选匹配最多的 → 拿不准列 2-3 个意图让用户选
